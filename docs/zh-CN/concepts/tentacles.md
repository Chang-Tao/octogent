# 触手

触手是 Octogent 中的核心抽象。

## 什么是触手

触手是 `.octogent/tentacles/<tentacle-id>/` 下的一个文件夹，用于存储代理可读的 markdown 文件。

最基本的有用文件包括：

- `CONTEXT.md`
- `todo.md`

允许添加额外的 markdown 文件，这些文件会在应用中以触手保险库文件的形式展示。

关键在于该文件夹是面向代理的。它是终端代理可以读取、编辑并交接给另一个终端的持久上下文。

## Deck 如何读取触手

Deck 不维护独立的触手上下文数据库副本。它扫描 `.octogent/tentacles/` 并从文件中派生大部分视图：

- 仅当文件夹包含 `CONTEXT.md` 时才被视为触手
- `CONTEXT.md` 中的第一个 `# 标题` 成为显示名称
- 该标题后的第一个非空段落成为描述
- 其他每个 `.md` 文件成为保险库文件，其中 `todo.md` 排在首位
- `todo.md` 中的复选框行成为进度和工作输入

Deck 特定的元数据（如颜色、状态、章鱼外观、路径和标签）单独存储在运行时状态中。这样可以将 UI 偏好与面向代理的 markdown 文件分开。

## 触手的用途

当代码库的某个切片或某条工作线需要自己的本地操作上下文时，使用触手。

示例：

- API 运行时
- 前端外壳
- 提示系统
- 监控集成
- 发布工作

## `CONTEXT.md` 的内容

`CONTEXT.md` 应说明：

- 该领域负责什么
- 重要的文件或目录
- 已经存在的内容
- 约束条件和边界情况
- 哪些内容不能破坏
- 对该触手特别有用的任何 Claude Code 技能（如适用）

第一个标题和第一个非空段落对运行时具有重要意义。请保持它们稳定且有用，因为它们将成为 Deck、Canvas、提示摘要和终端创建流程中显示的名称和描述。

当一个触手有建议的 Claude Code 技能时，Octogent 会在 `CONTEXT.md` 底部追加一个受管区块：

```md
<!-- octogent:suggested-skills:start -->
## 建议技能

如果需要，你可以使用这些技能。

- `skill-name`
<!-- octogent:suggested-skills:end -->
```

当建议技能发生变化时，API 会重写该受管区块。请将人工编写的架构说明放在该区块之外。

## `todo.md` 的内容

`todo.md` 应包含 markdown 复选框项：

```md
# 待办事项

- [ ] 为监控配置添加请求验证
- [ ] 在测试中覆盖无效负载情况
- [x] 将路由接入请求处理器
```

运行时会解析复选框行并计算进度。

只有匹配 `- [ ] text` 或 `- [x] text` 的行被视为待办项。它们的顺序很重要，因为集群创建在创建工作代理终端 ID（如 `<tentacle-id>-swarm-0`）时会使用解析后的项索引。

当 UI 切换、编辑、添加或删除待办项时，它会重写 `todo.md`。没有隐藏的待办存储。

## 触手与委派

触手的意义不仅在于文档，它还代表操作上下文。

挂载到触手的工作代理可以：

- 首先读取本地备注
- 保持限定在该领域内
- 将待办列表用作工作队列
- 将工作交给子代理而无需从头重建上下文

当从 Deck 解决一个待办项时，Octogent 会读取该项文本，解析工作代理提示模板，并创建一个挂载到同一触手的终端。对于集群运行，未完成的待办项成为工作代理，较大的集群会获得一个父协调终端来监督完成。

## 触手与工作树

触手与工作树不是同一个概念。

- 触手是一个上下文文件夹
- 工作树是为终端提供的独立 git 检出

你可以将触手与共享工作空间终端或工作树终端一起使用。

在共享模式下，所有终端在主工作空间中操作，因此上下文边界是约定性和流程性的。在工作树模式下，每个终端可以在 `.octogent/worktrees/` 下获得独立的检出，但它仍然读取同一个触手文件夹以获取指令和待办事项。

## 故障边界

- 删除触手会移除其面向代理的文件，但终端记录是独立的运行时状态
- 重启 API 会保留终端元数据和转录，但不会保留实时 PTY
- 通道消息不会持久化，因此持久的交接应放在触手 markdown 中
- 工作树分支与工作树支持的终端绑定，而非触手文件夹本身
